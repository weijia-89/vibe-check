# PROMPT_CALIBRATION_AGENT

Token-optimized prompt for vibe_check.py baseline calibration via stratified PR sampling.

## Usage

Copy the prompt below into your agent session. The agent must have:
- GitHub MCP tools (list PRs, get PR details, fetch file contents)
- Shell/bash for running gh CLI commands
- vibe_check.py in the working directory
- Write access to output directory

---

```
# VIBE_CHECK BASELINE CALIBRATION AGENT

GOAL: Sample statistically representative PRs from repo. Compute per-language & per-PR-type baselines for 10 vibe_check.py signals. Detect threshold differences.

## PHASE 1: LANGUAGE PROFILE

1. Enumerate all files in default branch: `gh repo view --json primaryLanguage`
2. For each file, extract extension. Count by extension.
3. For each extension, get total lines: `find . -name "*.EXT" -exec wc -l {} + | tail -1`
4. Map extensions to languages:
   - .py → Python, .js/.jsx/.ts/.tsx → JavaScript, .java → Java, .go → Go, .rs → Rust, .cpp/.cc → C++, .c → C, .cs → C#, .rb → Ruby, .php → PHP, .swift → Swift, .kt → Kotlin, .scala → Scala
5. Compute % by count and % by line count.
6. Output: language_profile.tsv with cols: lang, file_count, pct_count, line_count, pct_lines

## PHASE 2: STRATIFIED SAMPLING DESIGN

STRATA:
- Size: <50, 50-200, 200-500, 500-2000, >2000 lines changed
- Type: bug_fix, feature, refactor, docs, test, config, hotfix (infer from labels, branch name, title)
- Author: sample across >= 10 unique authors
- Time: recent 6-12 months (via `gh pr list --search "merged:>=2025-04-16" --limit 1000`)

BUDGET CONSTRAINTS:
- Max API calls: 5000
- Max PRs to process: 500
- Max time: 30 min wall clock
- Decrement budget per API call (1 per pr list, 5 per pr detail fetch)

PILOT PHASE (N=30):
1. Fetch 30 merged PRs uniformly at random from last 12 months.
2. For each: get diff, parse files, infer variant, run vibe_check.py signals.
3. Compute variance of each signal. Store in pilot_stats: {signal: var}.

STATISTICAL SIZING:
- Effect size threshold: Cohen's d = 0.5
- Power: 0.80
- Compute required N for each signal using formula:
  N = 2 * ((z_alpha + z_beta) / d)^2 where z_alpha=1.96 (α=0.05), z_beta=0.84 (β=0.20)
  For each signal: N_i = 2 * ((1.96 + 0.84) / 0.5)^2 ≈ 31 (worst case, adjust by observed SD)
- N_required = max(N_i) across signals, capped at 500.
- Adaptive: if variance is lower than expected, N drops. If higher, N increases but caps at 500.

ANTI-CASCADE GUARDS:
- If budget < cost of next batch: STOP and summarize available sample.
- Never recursively expand searches. One cycle only: pilot → estimate N → sample to N → stop.
- If N > 500, set N = 500 and flag in summary.

## PHASE 3: STRATIFIED PR SAMPLING

1. Use `gh pr list --search "is:merged merged:>=2025-04-16" --limit 500 --json number,title,labels,baseRefName,headRefName,additions,deletions,author` to enumerate.
2. For each PR:
   - Extract size = additions + deletions.
   - Map to size stratum.
   - Infer type:
     * Check labels: if any of {bug, fix, bugfix} → bug_fix. If {feature, feat} → feature. If {refactor} → refactor. If {docs, doc} → docs. If {test} → test. If {config, ci, deployment} → config. If branch contains "hotfix/" → hotfix.
     * Else, parse title for keywords: "fix/fixed" → bug_fix, "add/feature/new" → feature, "refactor" → refactor, "doc/docs" → docs, "test" → test.
   - Track author for diversity.
   - Track merged_at for time stratification.
3. Stratified random sample:
   - Aim for >= 3 PRs per (size, type) pair.
   - Sample target N across all strata proportionally to stratum size.
   - Ensure >= 10 unique authors.
4. Fetch selected PRs: `gh pr view <number> --json files,commits,title,labels,body`
5. For each PR: download diff, extract file list, detect languages.

## PHASE 4: SIGNAL COMPUTATION

For each sampled PR:
1. Checkout PR branch (or apply patch): `gh pr checkout <number>`
2. Run vibe_check.py on changed files (diff context only).
3. Extract 10 signals:
   - CCR: comments / non-blank code
   - Docstring %: count(functions with docstrings) / count(total functions)
   - Naming uniformity: % of identifiers matching dominant case (detect camelCase vs snake_case per language)
   - Error patterns: count(empty catch, log-only, blanket except) / count(total exception handlers)
   - Declarative bias: (assignments + returns) / (if + for + while + switch)
   - Func length: mean, coefficient of variation
   - Comment phrasing: % of comments matching AI phrases ("generated by", "auto-formatted", "assisted")
   - Hallucinated APIs: count(non-existent method calls) [requires language-specific stub DB]
   - Edge case depth: max nesting depth, null check density (checks per line)
   - Commit metadata: count(commit messages with AI markers: "GPT", "Claude", "Copilot", "AI", "generated") / count(total commits in PR)
4. Infer PR language from file mix: dominant language by LOC in diff.
5. Record: {pr_number, type, size_stratum, lang, ccr, docstring_pct, naming_uniform_pct, error_pattern_rate, decl_bias, func_len_mean, func_len_cv, comment_phrase_pct, halluc_api_count, edge_case_depth, commit_ai_marker_pct}

## PHASE 5: BASELINE COMPUTATION

Aggregate by language, type, overall:
- Per-signal: mean, median, std, IQR (q25, q75), p5, p95
- Per-language: separate rows for each signal & language combo
- Per-type: separate rows for PR type
- Overall: one row per signal across all PRs

Output: calibration.json with structure:
```json
{
  "metadata": {
    "repo": "<repo_url>",
    "sample_size": N,
    "sample_date": "2026-04-16",
    "languages": ["Python", "JavaScript", ...],
    "time_period": "2025-04-16 to 2026-04-16"
  },
  "baselines": {
    "overall": {
      "ccr": {"mean": X, "median": X, "std": X, "q25": X, "q75": X, "p5": X, "p95": X},
      ...
    },
    "by_language": {
      "Python": {"ccr": {...}, ...},
      ...
    },
    "by_type": {
      "bug_fix": {"ccr": {...}, ...},
      ...
    }
  },
  "variant_distribution": {type: count, ...},
  "author_count": <int>,
  "notes": "<brief findings>"
}
```

Output: sampling_log.tsv with cols: pr_number, type, size_stratum, lang, ccr, docstring_pct, naming_uniform_pct, error_pattern_rate, decl_bias, func_len_mean, func_len_cv, comment_phrase_pct, halluc_api_count, edge_case_depth, commit_ai_marker_pct

## PHASE 6: FINDINGS SUMMARY

Compute and report (< 200 words):
- Language-specific signal outliers (signal X in language Y differs from overall by > 1 SD)
- PR-type patterns (e.g., bug fixes have higher error_pattern_rate)
- Undersampled strata (warn if any stratum has N < 3)
- Recommended thresholds for flagging: if signal X differs from baseline by > 2 SD, candidate for alert.
- Any edge cases encountered (e.g., unsupported file types, API limits hit)

## EXECUTION CHECKLIST

- [ ] PHASE 1: Language profile generated
- [ ] PHASE 2: Pilot sample (N=30) complete, variance estimated, N_required computed
- [ ] PHASE 3: Stratified sample collected (N=N_required, capped at 500)
- [ ] PHASE 4: All signals computed for all PRs
- [ ] PHASE 5: Baselines aggregated, JSON + TSV written
- [ ] PHASE 6: Summary written

## OUTPUT FILES

- calibration.json
- sampling_log.tsv
- language_profile.tsv
- summary.txt (< 200 words)

STOP after writing outputs. Do not proceed to threshold tuning; that is a separate workflow.
```

---

## Notes for Agent Runner

- The prompt assumes vibe_check.py is available and executable with syntax: `python vibe_check.py <file_or_dir> --json`
- The agent must implement variance calculation; recommend bootstrap CI (resample 1000x, compute 95% credible interval for each signal).
- If API rate limits are hit, log and continue with available sample (don't cascade).
- Language detection is critical; if a PR touches multiple languages, stratify the analysis by the dominant language in the diff.
